Integration Guide

Transportation & Real-Time Tracking

Welcome to the live tracking developers portal. This guide details REST APIs, dynamic Socket.IO routing event cycles, and rules configured for passenger-driver live telemetry.

1. System Overview

Architecture and real-time streaming components

HTTP

REST Controller

Exposes endpoints for fleet setup, patient booking generation, trip start verification, and dispatch controls.

IO

Socket.IO Core

Handles high-concurrency GPS ingestion stream, rooms connection hooks, and passenger coordinate broadcasts.

DB

Redis & Prisma

Keeps current asset coordinates and vehicle availability states synchronized in low-latency memory.

2. Real-Time Telemetry via Socket.IO

Socket events description and payload specs

DRIVER CONN Driver Connection & Handshake
Query Handshake

To receive incoming ride dispatches, the **Driver App** must pass its encrypted driver profile ID as a query parameter during the socket handshake. The server automatically decrypts the ID and adds the driver socket to their private room: driver_room:<decryptedDriverId>. 

// 1. Driver App Connection Handshake
const socket = io("http://localhost:5000", {
  query: {
    driverId: "drv_WtA2bsgxL5enW2136A9sezEc..." // Encrypted driver public ID
  }
});

// 2. Server implementation (socket.ts)
// const driverId = socket.handshake.query.driverId;
// const rawDriverId = decryptPublicId(driverId);
// socket.join(`driver_room:${rawDriverId}`);
USER SUB subscribeTripRoom
Inbound Socket Event

To receive real-time telemetry updates for an assigned vehicle, the **Passenger App** connects to the socket server (without query params) and emits subscribeTripRoom passing the encrypted tripId (the booking master ID). The server subscribes the socket to trip_room:<tripId>.

// 1. Passenger App Connection
const socket = io("http://localhost:5000");

// 2. Join telemetry stream for an active booking
socket.emit("subscribeTripRoom", {
  tripId: "bkm_S4fIR4U92p6SEc_XBx..." // Encrypted Booking Master ID
});

// 3. Listen to joining status
socket.on("subscribed", (status) => {
  console.log("Subscribed successfully to:", status.room); // trip_room:bkm_S4fIR4U92...
});
JOIN join
Inbound Socket Event

Allows any client to explicitly join any custom named room for custom broadcast topics.

// Join a general custom room
socket.emit("join", {
  room: "notification_alerts"
});
EMIT emitTelemetryUpdate
Inbound Event

Emitted by the **Driver App** to feed real-time coordinates. The backend checks movement rules before executing database updates.

Telemetry Ingestion Ruleset:

  • Ingested immediately if it is the driver's first packet.
  • Saves/broadcasts if elapsed time >= 5 seconds.
  • Saves/broadcasts if linear offset movement is >= 20 meters (approx. 0.00018 degrees).
{
  "driverId": "string (Encrypted Driver Profile Public ID)",
  "tripId": "string (Encrypted Booking Master Public ID)",
  "lat": 12.9716,
  "lng": 77.5946
}
BROADCAST telemetryUpdate
Outbound Event

Broadcasted by the server to all users in the corresponding `trip_room: ` room.

{
  "driverId": "string (Encrypted Driver Public ID)",
  "tripId": "string (Encrypted Booking Master ID)",
  "latitude": 12.9716,
  "longitude": 77.5946,
  "timestamp": "2026-06-08T11:00:00.000Z"
}

Identity Resolution — How userId Is Picked

When no token is present, the controller resolves identity using a priority chain. There are two separate chains — one for regular users and one for drivers.

Frontend User 
const userId =
  req.user?.userId       // 1. Real token (production)
  ?? req.body.userId     // 2. Pass in body for testing
  ?? 'TEST_USER_ID';     // 3. Auto fallback

Used by: createBooking, createTransportationRequest, getUnifiedBookings, cancelRequest, verifyBooking, extendBooking

Driver 
const userId =
  req.user?.userId       // 1. Real token (production)
  ?? req.body.driverId   // 2. Pass driverId in body
  ?? req.body.userId     // 3. Generic userId fallback
  ?? 'TEST_DRIVER_ID';   // 4. Auto fallback

Used by: acceptBooking, toggleDriverOnline

Per-Endpoint Override Reference

To test with a real database record, pass the override field in your request. Omit it to use the generic fallback.

/vehicle/bookings/:id/verify POST User body.userId TEST_USER_ID

/vehicle/bookings/:id/accept POST Driver body.driverId TEST_DRIVER_ID

POST /api/v1/patients Add a family member

Creates a new family member linked to the logged-in user. SELF relation is not allowed. Rejects duplicates with the same firstName + lastName + relation combination. Returns the created record with profile completeness status.

gender enum optional MALE | FEMALE | OTHER

GET /api/v1/patients/:id/completeness Check profile completeness

Returns the profile completeness status of a specific patient before confirming a booking. The required fields for a COMPLETE status are: firstName, lastName, age (or dateOfBirth), bloodGroup, mobileNumber, and relation.

Request 
GET /api/v1/patients/enc_XyZ789.../completeness
Authorization: Bearer <token>
:id is the encrypted patient ID from the GET /patients response.
200 OK — Response 
{
  "success": true,
  "message": "Patient completeness retrieved successfully",
  "data": {
    "id": "enc_XyZ789...",
    "name": "Anita Kumar",
    "relationship": "CHILD",
    "profile_status": "INCOMPLETE",
    "missing_fields": ["bloodGroup", "mobileNumber"],
    "fields": {
      "firstName": "Anita",
      "lastName": "Kumar",
      "age": 10,
      "dateOfBirth": "2015-06-15",
      "bloodGroup": null,
      "mobileNumber": null,
      "relation": "CHILD"
    }
  }
}

COMPLETE

All required fields are present. Patient is ready for booking.

INCOMPLETE

One or more required fields are missing. Use PATCH /patients/:id to fill them before booking.

PATCH /api/v1/patients/:id Update patient profile

Partially updates a patient's profile. Only the provided fields are updated — omit any field you don't want to change. Updates both the FamilyMember and Patient database records in a single transaction. Returns the full updated profile with completeness status.

Field Type Updates Notes
firstName string FamilyMember + Patient.name Also updates the combined display name
lastName string FamilyMember + Patient.name Also updates the combined display name
age integer FamilyMember + Patient Stored on both records
dateOfBirth string (date) FamilyMember Auto-calculates and overwrites age in FamilyMember
bloodGroup string FamilyMember e.g. O+, AB-
mobileNumber string FamilyMember 10-digit mobile number
gender enum Patient MALE | FEMALE | OTHER
